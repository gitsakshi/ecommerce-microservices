🎯 Interview Question

“Design a Rate Limiter for an API.”

Goal:
Limit how many requests a user can make.

Example:

100 requests per minute per user

✅ Step-by-Step Answer (Use this structure)
⭐ 1️⃣ Clarify Requirements

Say:

“Is this per user, per IP, or per API key?”

Assume:

Per user
100 requests/min

⭐ 2️⃣ High-Level Idea

Say:

“We need to track request counts and block when limit exceeds.”

Flow:

Client → API Gateway → Rate Limiter → Service


Rate limiter sits before main service.

⭐ 3️⃣ Where to Store Counters?

Best answer:

“Use Redis because it’s fast and in-memory.”

Why?

low latency

supports atomic operations

TTL support

⭐ 4️⃣ Logic (Simple)

For each request:

Check user counter in Redis
If count < 100 → allow
Else → block

⭐ 5️⃣ Algorithm Options

Mention 1–2 (not all):

Fixed Window (easiest)
100 requests per minute window
counter resets every minute

Token Bucket (better)
tokens refill gradually
smoother traffic


👉 Even naming these scores points.

⭐ 6️⃣ Scaling

Say:

“Redis can be shared across multiple servers.”

So rate limit works in distributed system.

⭐ 7️⃣ Failure Handling

Say:

“If Redis is down, we can allow requests to avoid blocking users.”

This shows real-world thinking.

✅ Interview-Ready Summary Answer

If short on time, say:

“I’d implement rate limiting at API Gateway level using Redis to store per-user counters with TTL. Each request increments a counter, and if it exceeds the threshold, we return 429 Too Many Requests. For smoother control, token bucket algorithm can be used.”

That’s a strong answer.
